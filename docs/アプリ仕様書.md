# 学習型・占いライティングアプリ「自己進化型コンテンツ生成エンジン」超詳細仕様書（Next.js + FastAPI スケール版）

最初からフロントエンドとバックエンドを完全に分離するモダンなアーキテクチャを採用することで、今後の機能拡張（ベクトルDBの本格導入やマルチユーザー化など）に耐えうる強固な基盤となります。

実装のたびに包括的なガイドを提供する方針に則り、APIエンドポイントの設計やディレクトリ構造の概念も含めた、より解像度の高い仕様書として再定義しました。

---

## ■ 1. コンセプト＆ビジョン

**「日々の仕事や家事、開発に疲れた人々の肩の力を抜き、心に余裕を生み出すコンテンツ生成エンジン」**

読者の不安を煽って依存させる「ダークパターン」を完全に排除し、心理学とカウンセリング技術（傾聴・共感・リフレーミング）を駆使して「体温のある文章」を生成・学習し続けます。特に「仕事と向き合っている社会人の方や、家事と向き合っている主婦の方々のリアルな悩み（一次データ）」を扱うことで、深く寄り添い刺さるコンテンツを提供します。

### 【倫理的・社会的ガイドライン（モラルルール）】

本アプリは「占い」や「カウンセリング技術」を謳う性質上、社会通念・法律上の問題を防衛するため、システム要件として以下のガイドラインを遵守します。

1. **医療・心理的アドバイスの禁止と免責:** 医療行為や専門的な心理療法を提供するものではありません。病気の診断、治療法の提示、生死に関する言及はプロンプトレベルで固く禁じ、システム内にも免責事項を明言します。
2. **占い分野特有の禁止事項:** 投資・ギャンブルの予測、犯罪の示唆、極度な不安を煽る表現（霊感商法等に繋がるもの）のAI生成をガードレールで厳重に禁止します。
3. **著作権と入力データの権利保護:** 参考記事として入力するテキストは、利用許諾を得ているもの、または著作権フリー・自作のものに限定することを運用上の大前提とします。システムの抽出ロジック設計において独自表現を完全に排除し、著作権侵害（翻案権等）を防ぐ構造とします。
4. **プライバシーと個人情報保護:** 入力される「悩み」のデータ等に個人情報（実名、所属企業、連絡先など）が含まれていた場合、システム側（またはLLMによる前処理）で自動的にマスク・除外処理を施しDBに保存させないようにします。
5. **AI生成コンテンツの透明性:** 生成された記事を外部プラットフォーム（noteなど）へ投稿する際は、各プラットフォームの「AI生成コンテンツに関する規約」を遵守し、AIによる生成支援を受けたことを明示する機能を投稿支援として推奨します。

## ■ 2. システム構成と技術スタック

フロントエンドとバックエンドを疎結合にし、RESTful API（またはGraphQL）で通信する構成です。

- **Frontend:** Next.js (App Router), TypeScript, Tailwind CSS
  - リッチで快適なUI/UXを提供。コンポーネント指向で保守性を高めます。
- **Backend:** FastAPI (Python), Pydantic
  - 高速なAPIサーバー。型安全なデータのやり取り（Pydantic）と、自動でのAPIドキュメント（Swagger UI）生成が強力です。
- **Database:** PostgreSQL + SQLAlchemy (ORM)
  - 最初からリレーショナルDBを採用し、データの整合性を担保。将来的なベクトル検索拡張（`pgvector`）への移行もシームレスです。
- **LLM API:** 軽量モデル (GPT-4o-mini 等) と 高精度モデル (Claude 3.5 Sonnet) の併用
  - コストとクオリティを両立するため、**データ構造・ナレッジ作成（構造化・抽出・DB化）には軽量LLM**を、最終的な**ブログ記事の長文生成には高精度なClaude 3.5 Sonnet**を使用するハイブリッド構成とします。

### 【システム構成フロー】

1. **データ自動収集パイプライン:** 運用者のUIトリガーにより、FastAPIがnoteのハッシュタグRSSから悩みデータを自動取得〜フィルタリングし、軽量LLM（GPT-4o-mini等）で構造化してDBへ蓄積。
2. **ユーザー操作の受付:** **Next.js (Client)** がユーザーから記事生成のオプションを受け付け、JSON形式で送信。
3. **RAGとプロンプト構築:** **FastAPI (Server)** がリクエストを受信。DBから過去のナレッジや基礎占術データを類似検索（ベクトル検索）で抽出。
4. **記事生成:** **FastAPI** がメインLLM（Claude 3.5 Sonnet）へ構造化されたプロンプトを送信し、高品位な記事をストリーミングで**Next.js**へレスポンス。

## ■ 3. 学習データセット（7本柱）とコア機能要件

本システムは、圧倒的な「刺さる文章」を生成するため、以下の7つの要素からなる**「自分専用データセット」**を核として機能します。

### 【データセットの構成要素】

1. **人気占いコンテンツからの構造分解（一次資料）**
   - **概要:** noteの上位有料占い記事、雑誌系（SPUR等）占いなどの「愛される構造」を分析します。
   - **システム機能:** 丸コピーは排し（❌）、システムの抽出エンジンにより「構造のみ」を抜き出しナレッジ化します（⭕️ 構造分解）。
2. **心理学・認知バイアス（「効き」の正体）**
   - **概要:** バーナム効果、確証バイアス、自己成就予言など、読者に「刺さる」メカニズムの体系化。
   - **システム機能:** これらを理解・定義した上で、記事生成プロンプトの基礎（システムプロンプトのルールやインコンテキストラーニング）として組み込みます。
3. **カウンセリング・コーチング技術（「温度」を作る）**
   - **概要:** 傾聴、共感、リフレーミング。占いを「軽いカウンセリング」として再定義します。
   - **システム機能:** 抽出した情報をデータベース上で「共感・傾聴」「視点の転換」のナレッジとして分類・管理し、生成時に利用します。
4. **コピーライティング（「深く刺さる」を決める）**
   - **概要:** 共感導入、ストーリーテリング、そして「悩みの手放し・自己受容（Relief）」。タイトルとテキスト構成に直結します。
   - **システム機能:** 一般的な「行動喚起（CTA）」は避け、読者が読んで肩の力が抜ける結びをフォーマット制約として必ず出力させます。
5. **社会人・主婦の一次データ（差別化の核）**
   - **概要:** 日々仕事と向き合う社会人や、家事・育児と向き合う主婦層が利用する特定のハッシュタグ（仕事辞めたい、人間関係の悩み、育児の疲れ等）から得られる生の声。
   - **システム機能:** 「ワンクリックお悩みデータ自動収集機能」により、note RSSから動的かつ自動で取得・構造化され、後述する「トピック自動提案機能」の種データとして絶えずDBに供給されます。
6. **基礎占術とマインドフルネス教義（ストーリーの骨格）**
   - **概要:** 西洋占星術（星回り・運命論）やタロット、また温かな文体のベースとなるマインドフルネスの教示などの基礎資料。
   - **システム機能:** 専門書やテキスト資料を読み込んでDB（ベクトル空間）化。記事生成において「悩みが些細に思える理由（運命論）」と「肩の力を抜く視点（マインドフルネス）」の根本ロジックとして参照（RAG）します。
7. **一貫した占い師ペルソナ（ファン形成の軸）**
   - **概要:** ファンがつく占い師やコンテンツに共通する特徴・口調・振る舞いを定義した特性データ。
   - **システム機能:** 人気コンテンツのデータ解析から「愛されるペルソナ特性」を導き出し、生成するすべての記事に一貫したキャラクター性を自然に盛り込みます。

---

### ① セーフティ＆ウォーム・ナレッジ抽出機能（構造分解エンジン）

既存の人気占い・良質な記事から、表面的な表現（著作物）を削ぎ落とし、「なぜこの文章が暖かいのか」という骨組みだけを抽出します。

- **入力:** 参考記事テキスト、または社会人・主婦層からの一次データ（各種SNSのポストなど。※権利処理済みテキストに限る）
- **制約:** 固有名詞・独自表現の完全排除。心理学・コーチング的な意図（抽象的なパラメーター）のみを抽出。個人情報（PII）のフィルタリング処置。
- **出力（構造化ナレッジ）:** ターゲットの悩み、共感導入フレーム、カウンセリング手法（リフレーミング等）、具体的なスモールステップ行動提案（Action）、癒やしと手放しの結び（Relief）。

### ② 基礎占術・思想ドキュメント解析機能（RAG基盤構築）

- **概要:** 西洋占星術、タロット占い、および「暖かく共感できる文体」のベースとなるマインドフルネス等の特定の信奉に関する資料を自律的に読み込み、DB化する機能。
- **入力:** 占術の解説テキスト、哲学・マインドフルネスの資料。
- **処理:** 軽量LLM（GPT-4o-mini等）を用いて「特定の星回り・カードが示す運命論的な意味」と「解決策として提示すべきマインドフルネス・共感の教え」への安価かつ高速な構造化を行い、ベクトル化して保管します。
- **出力:** `knowledge_astrology` および `knowledge_mindfulness` データベースへの蓄積。

### ③ 占い師ペルソナ形成＆トピック自動サジェスト機能

- **ペルソナの一貫性構築:** ナレッジベースの分析結果から、ファンがつく口調や振る舞い（ペルソナ特性）を定義し、システム内で一貫して保持します。
- **トピック自動提案:** 蓄積されたDB（仕事や家事�\* **`/` (Dashboard):** 記事数、平均スコア、人気テーマのサマリー表示に加え、**「悩みデータを自動収集」ボタン**、および現在の `worry_seeds` ・トピック候補数のサマリーを表示。
- **`/collect` (悩み収集履歴):** `worry_seeds` テーブルの内容一覧（収集日時・タグ・悩み内容・変換済み法）を確認・手動削除できる画面。「何が元データになっているか」を運用者が把握するために重要。
- **`/documents` (基礎資料管理):** 西洋占星術やマインドフルネスの資料をアップロードし、RAG用DBに変換・管理する画面。
- **`/extract` (ナレッジ抽出):** テキストエリアに参考記事を入力し、APIを右いて結果をプレビュー＆保存する画面。
- **`/knowledge` (ナレッジ管理):** 蓄積されたナレッジやペルソナ特性をテーブル表示、編集する画面。
- **`/generate` (トピック選択＆記事生成):** システムが提案する「ブログトピック候補」一覧から書きたい内容を選択（または手動入力）し、AIが生成する本文をリアルタイム（Streamingレスポンス）で逐次プレビュー表示する画面（タイムアウト防止UX）。
- **`/articles` (記事・投稿管理):** 生成済み記事の一覧。編集、クリップボードへのコピー、および投稿後のフィードバック（スキ・コメント数）を入力する画面。
  1. **悩みの肯定（西洋占星術等の運命論）:** 挫折や問題の原因を「星回り」や「運命」の解釈に委ね、読者を自己責任論から解放し肯定する。
  2. **肩の力を抜く（マインドフルネス・癒やし）:** 「悩みがどうでもよくなる」「暖かい気持ちになり余裕が生まれる」ような、執着を手放す視点の転換を提示する。
  3. **一貫したペルソナの注入:** 定義された占い師ペルソナの口調・スタンスを自然に記事全体へ適応させる。
  4. **コピーライティング:** 上記ステップを「共感導入→ストーリー展開→悩みの手放し（Relief）」の型に乗せて出力し、無理な行動喚起（CTA）は行わない。
  5. **スモールステップ提案（Action）:** 記事の締めに `knowledge_action` から状況に最も合致するスモールステップ（自己受容・自愛につながる具体的な行動）をRAGで1〜2件引いて温かく添える。
- **制約（ガードレール）:** 断定的な未来予測の禁止、医療・メンタルヘルスの断定禁止、ギャンブル・法的アドバイスの禁止、焦燥感を煽る表現の禁止。心理学バイアスを悪用せず、読者が「肩の力が抜け、心に余裕を持てる」心理的安全性の提供にのみ利用すること。
- **出力:** 記事の構成案、タイトル案、本文。

### ⑤ ナレッジ・データベース管理＆自己進化フィードバック機能

- **フィードバック進化:** note投稿後の「スキ」「コメント数」などの反響をUIから手動入力。高スコアのナレッジを分析し、次回生成時の検索ウェイトを引き上げます。

### ⑥ ワンクリックお悩みデータ自動収集機能（RSSスクレイピング＆軽量LLMパイプライン）

- **トリガー（UI）:** 管理画面（ダッシュボード）の「悩みデータを自動収集」ボタンを非同期バッチ処理のトリガーとして配置。
- **取得元と動的パラメータ生成:** `https://note.com/hashtag/[ハッシュタグ名]/rss` を取得元とし、ハッシュタグはバックエンド側で定義した「一般の悩みに関連するリスト（例：仕事辞めたい、人間関係の悩み、育児の疲れ、将来の不安など）」からランダムまたはローテーションで自動選択する。
- **期間の動的フィルタリング:** RSS結果から「過去1週間〜1ヶ月の間」のデータのみを抽出（直近すぎ・古すぎを排除しトレンドを押さえるため）。対象期間のデータが不足する場合は、検索ハッシュタグ数を動的に増やして必要件数を確保する。
- **処理フロー（全4ステップを非同期バッチとして一本化）:**
  1. **取得:** noteのRSSから指定ハッシュタグ記事を取得・期間フィルタリング。
  2. **プレスクリーニング:** 軽量LLMに「この記事は未解決の悩みを書いた一次資料か？（yes/no）」を分類させ、`no` の記事（広告・解決済み体験談等）を除外。コスト：1記事あたり数十トークンで完了。
  3. **抽象化→`worry_seeds`保存:** 通過した記事を固有名詞排除・抽象化して `worry_seeds` へ保存（`is_used=False`）。
  4. **トピック自動生成:** 保存した `worry_seeds` を元に軽量LLMが `blog_topics_candidates` を自動生成（`is_used=True` に更新）。
  - 生データは変換後に破棄する。`GET /api/topics/candidates` は既存DBから返すだけにしてレイテンシゼロを保証する。
- **【コスト爆発防止のガードレール】:** LLMのAPI費用暴騰を防ぐため、以下の制約システムを設ける。
  1. **実行上限のハードリミット:** 1回の収集ボタン押下でLLMへ送信するスクレイピング記事数は「最大〇件（例: 20件）」に制限する。
  2. **トークン切り捨て:** RSSから取得したテキストが極端に長い長文記事だった場合、先頭〇〇〇トークンのみをLLMに送信し、残りは強制カットする。
  3. **モデルの強制固定:** この収集パイプラインに関する処理は、必ず最も安価な推論モデル（GPT-4o-mini等）を固定で使用する。

## ■ 4. データベース設計（PostgreSQL / SQLAlchemy モデル定義）

**1. `articles` (生成・投稿記事)**

- `id` (UUID, Primary Key)
- `title` (String)
- `content` (Text)
- `status` (String: 'draft' | 'posted')
- `topic_id` (UUID, Nullable: 手動テーマ入力の場合はNULL)
- `topic_id` (UUID, Nullable: 手動テーマ入力の場合はNULL)
- `note_url` (String, Nullable)
- `likes` (Integer, Default 0)
- `comments` (Integer, Default 0)
- `score` (Float, Default 0.0)
- `used_knowledge_ids` (JSONB: 使用したナレッジIDの配列)
- `theme` (String: 生成時に使用したトピックテーマ)
- `created_at`, `updated_at` (DateTime)

**2. `knowledge_empathy` (共感・傾聴・カウンセリング)**

- `id` (UUID, Primary Key)
- `abstract_situation` (String: 社会人・主婦の悩み等、抽象化された状況)
- `empathy_approach` (Text: 寄り添い・傾聴のプロセス)
- `warmth_level` (Integer: 1-5)
- `embedding` (Vector: 対象の悩みに最適な共感ロジックを引くためのベクトル)
- `created_at`, `updated_at` (DateTime)

**3. `knowledge_reframing` (視点の転換)**

- `id` (UUID, Primary Key)
- `negative_state` (String)
- `positive_perspective` (Text: 心理学を活用したポジティブな視点)
- `embedding` (Vector: 対象の悩みに対するリフレーミング検索用)
- `created_at`, `updated_at` (DateTime)

**4. `knowledge_action` (癒やしの行動提案)**

- `id` (UUID, Primary Key)
- `situation` (String: 特定の疲れや悩み)
- `small_step_proposal` (Text: 「今日は温かいお茶を飲んで寝る」等の自愛を促す具体的なスモールステップ)
- `created_at`, `updated_at` (DateTime)

**5. `knowledge_copywriting` (コピーライティングと癒やしの結び)**

- `id` (UUID, Primary Key)
- `structure_type` (String: 'Introduction' | 'Storytelling' | 'Relief')
- `suggested_text_pattern` (Text: 刺さる文章の型・心の余裕を生む手放しのパターン)
- `created_at`, `updated_at` (DateTime)

**6. `analysis_results` (分析結果)**

- `id` (UUID, Primary Key)
- `winning_patterns` (JSONB: 高スコア記事の共通パターンを構造化したデータ)
- `created_at`, `updated_at` (DateTime)

> **運用フロー:** `PATCH /api/articles/{article_id}/feedback` で高スコアが付いた記事が一定数を超えた段階で、バックグラウンドJobが `articles` テーブルの高スコアデータを分析し、優れた記事の文体構造・使用ナレッジ傾向などを `winning_patterns` として書き込む。次回以降の生成プロンプト構築時に参照される。

**7. `knowledge_astrology` (基礎占術・運命論DB)**

- `id` (UUID, Primary Key)
- `astrology_type` (String: 西洋占星術, タロット等)
- `theme_or_symbol` (String: 星座、惑星、カード名等)
- `fatalistic_meaning` (Text: 運命論的な解釈・悩みの肯定ロジック)
- `embedding` (Vector: pgvector等を用いた類似検索用)
- `is_seasonal` (Boolean, Default False: 特定の季節・時期に旬となる内容かどうか)
- `valid_from` (String, Nullable: 例 "03-01" など月日形式。この日付以降に優先参照)
- `valid_until` (String, Nullable: 例 "03-31"。この日付まで優先参照)
- `created_at`, `updated_at` (DateTime)

**8. `knowledge_mindfulness` (マインドフルネス・思想DB)**

- `id` (UUID, Primary Key)
- `philosophy_concept` (String: 思想の概念)
- `encouragement_text` (Text: 温かさ、共感、自己肯定を促す文体サンプルや解決策)
- `embedding` (Vector: pgvector等を用いた類似検索用)
- `created_at`, `updated_at` (DateTime)

**9. `knowledge_persona` (占い師ペルソナDB)**

- `id` (UUID, Primary Key)
- `trait_name` (String: ペルソナ特性の名前。例：包容力, 知的, ユーモア)
- `trait_description` (Text: どう振る舞うか、語尾やスタンスの定義)
- `is_active` (Boolean: 現在適用中かどうか)
- `priority_order` (Integer, Default 0: 複数のペルソナ特性を order 順に結合しシステムプロンプトへ注入。数値が小さいほど優先。これにより包容力＋ユーモアなど複数特性のブレンドが可能)
- `created_at`, `updated_at` (DateTime)

**10. `worry_seeds` (お悩みの種プールDB)**

- `id` (UUID, Primary Key)
- `original_text_hash` (String: 重複登録防止用のハッシュ)
- `abstract_worry` (Text: 軽量LLMにより抽象化された悩みデータ)
- `source_tags` (String: 収集元のハッシュタグ等)
- `is_used` (Boolean, Default False: `blog_topics_candidates` まで変換済みかどうか)
- `created_at`, `updated_at` (DateTime)

**11. `blog_topics_candidates` (トピック候補DB)**

- `id` (UUID, Primary Key)
- `topic_title` (String: トピック案)
- `source_data_ids` (JSONB: 根拠となった `worry_seeds.id` 等)
- `status` (String: 'suggested' | 'used' | 'dismissed')
- `priority` (Integer, Default 0: 大きいほど優先表示される。悩みの発生頻度・スコア期待値から自動計算。)
- `created_at`, `updated_at` (DateTime)

## ■ 5. API設計（FastAPI エンドポイント仕様）

- **`POST /api/collect/note`**
  - Req: `{}` (パラメータ不要。バックエンド側で自動構成)
  - Res: `{ "status": "processing", "message": "Background task started" }` (非同期実行での収集開始)
- **`POST /api/documents/parse`**
  - Req: `{ "type": "astrology" | "mindfulness", "content": "資料テキスト..." }`
  - Res: 資料を構造化・ベクトル化してDB化（RAG用基盤へのインサート）
- **`POST /api/extract`**
  - Req: `{ "text": "参考記事テキスト" }`
  - Res: 抽出されたナレッジのJSON（この時点ではDB保存せずプレビュー用）
- **`POST /api/knowledge`**
  - Req: 抽出プレビューで確認したナレッジデータ
  - Res: `{ "status": "success", "ids": [...] }` (DB保存実行)
- **`GET /api/knowledge/{type}`**
  - Res: 指定タイプ(empathy, reframing, action, copywriting, persona)のナレッジ一覧取得
- **`GET /api/topics/candidates`**
  - Res: DBから自動生成された予測ブログトピックの一覧
- **`POST /api/generate`**
  - Req: `{ "topic_id": "選択したトピックID", "theme": "テーマ（カスタマイズ用）" }`
  - Res: `text/event-stream` （Server-Sent Events）形式で生成テキストをストリーミング返却。完了時に `{ "title": "..." }` イベントを送信。
- **`POST /api/articles`**
  - Req: 生成した記事の保存データ
  - Res: `{ "article_id": "..." }`
- **`POST /api/articles/{article_id}/regenerate`**
  - Req: `{ "change_persona": true/false, "custom_theme": "..." }` (オプション。ペルソナ変更・テーマ上書きを指定可能)
  - Res: `text/event-stream` （Server-Sent Events）形式でストリーミング。同じ topic_id を再利用し status を 'draft' に戻す。
- **`PATCH /api/articles/{article_id}/feedback`**
  - Req: `{ "likes": 10, "comments": 2 }`
  - Res: 再計算されたスコア情報

**【共通エラー・セキュリティ仕様】**

- 各エンドポイントはLLMからの不適切な出力エラーや、外部APIのレートリミットエラー（`429 Too Many Requests`）に対する適切なフェイルセーフとログ記録を実施します。
- 脆弱性対策として、入力値のサニタイズ（XSS/SQLインジェクション防止）をフレームワークの標準機能（Pydantic等）で徹底します。
- 将来的なマルチユーザー化を見据え、CORS（Cross-Origin Resource Sharing）の適切な設定と、システム連携時のAPIアクセスにおける簡易的な認証（APIキー等）を初期段階から見込みます。

## ■ 6. UI構成（Next.js App Router）

- **`/` (Dashboard):** 記事数、平均スコア、人気テーマのサマリー表示に加え、システムに記事を収集させる**「悩みデータを自動収集」ボタン**を配置。
- **`/documents` (基礎資料管理):** 西洋占星術やマインドフルネスの資料をアップロードし、RAG用DBに変換・管理する画面。
- **`/extract` (ナレッジ抽出):** テキストエリアに参考記事を入力し、APIを叩いて結果をプレビュー＆保存する画面。
- **`/knowledge` (ナレッジ管理):** 蓄積されたナレッジやペルソナ特性をテーブル表示、編集する画面。
- **`/generate` (トピック選択＆記事生成):** システムが提案する「ブログトピック候補」一覧から書きたい内容を選択（または手動入力）し、AIが生成する本文をリアルタイム（Streamingレスポンス）で逐次プレビュー表示する画面（タイムアウト防止UX）。
- **`/articles` (記事・投稿管理):** 生成済み記事の一覧。編集、クリップボードへのコピー、および投稿後のフィードバック（スキ・コメント数）を入力する画面。

## ■ 7. 包括的実装ガイドライン（フェーズ別実行計画）

この構成を最短かつ確実に行うための実装ステップです。

### Phase 1: FastAPIの基盤構築とDB連携

まずはバックエンド単体でデータが正しく保存・取得できる状態を作ります。

1. Python仮想環境の構築と `fastapi`, `uvicorn`, `sqlalchemy`, `psycopg2-binary`, **`pgvector`** のインストール。
2. PostgreSQLのセットアップ（Dockerコンテナでの立ち上げを推奨）。起動後に `CREATE EXTENSION IF NOT EXISTS vector;` を必ず実行しpgvectorを有効化。
3. SQLAlchemyを用いたモデル定義と、マイグレーション（Alembic）の設定。
4. 基本的なCRUDを行うエンドポイント（`GET /api/knowledge`, `POST /api/articles` など）の実装とSwagger UI（`/docs`）での動作確認。

### Phase 2: LLMコアエンジンのAPI統合

FastAPI内に、LLMと通信するビジネスロジックを実装します。

1.  LLMクライアント（Anthropic SDK / OpenAI SDK）の組み込み。
2.  `POST /api/collect/note`（自動収集）と `worry_seeds` への保存ロジック実装。ガードレール（件数上限・トークンカット）の動作確認。
3.  `GET /api/topics/candidates`: `worry_seeds` → `blog_topics_candidates` 自動生成フローの実装。
4.  `POST /api/extract`: 制約（ダークパターン排除・著作権保護）を組み込んだシステムプロンプトの作成と、構造化出力（JSON）のパース処理。
5.  `POST /api/generate`: DBから全6種ナレッジ（empathy, reframing, action, astrology, mindfulness, persona）をRAGで取得し、コンテキストに埋め込んで記事をストリーミング生成する処理。

### Phase 3: Next.jsフロントエンドの構築

完成したAPIと連携するUIを構築します。

1.  `npx create-next-app` によるプロジェクト初期化（Tailwind CSS導入）。
2.  共通レイアウト（サイドバーナビゲーション等）の作成。
3.  各画面（`/extract`, `/generate`, `/articles`）のUI実装。
4.  `fetch` または `axios`, `SWR`/`React Query` を用いたFastAPIとの通信処理の実装。

### Phase 4: 自己進化フィードバックループの完成

1.  Next.js側からのフィードバック数値送信UIの実装。
2.  FastAPI側でのスコア計算ロジック（例: スキ数とコメント数に重みをつけた算出）の実装。
3.  生成エンドポイント（`POST /api/generate`）の改修：スコアの高いナレッジをDBから優先的にフェッチし、LLMのプロンプトに組み込む処理の追加。
