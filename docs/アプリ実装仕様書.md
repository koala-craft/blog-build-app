## ■ 0. LLM接続構成 (OpenRouter 互換モード)
開発初期は OpenRouter の無料モヂルを活用します。
OpenRouter は OpenAI 完全互換 API を提供しているため、openai SDK の ase_url を切り替えるだけで接続可能です。

* **Base URL:** https://openrouter.ai/api/v1
* **API Key:** OPENROUTER_API_KEY (無料モデルの場合も必要)
* **デフォルト無料モデル:**
  * 抽出用: google/gemini-2.0-flash-lite-preview-02-05:free
  * 執筆用: deepseek/deepseek-r1:free

`python
# 実装イメージ (backend/app/core/llm.py)
from openai import AsyncOpenAI
client = AsyncOpenAI(
    base_url=settings.LLM_BASE_URL,
    api_key=settings.OPENROUTER_API_KEY
)
`

## 1. テクノロジースタックと依存関係

### フロントエンド (Client)
* **Framework:** Next.js 14+ (App Router採用)
* **Language:** TypeScript
* **Styling:** Tailwind CSS (デザインシステム構築用)
* **State Management/Data Fetching:** SWR もしくは React Query, および標準の `fetch`
* **UI Libraries:** Lucide React (アイコン群), React Markdown (生成された記事のプレビュー用)

### バックエンド (Server)
* **Framework:** FastAPI
* **Language:** Python 3.11+
* **Database ORM:** SQLAlchemy (v2.0+)
* **Database Driver:** `psycopg2-binary` または `asyncpg`
* **Vector DB/Search:** PostgreSQL + `pgvector` 拡張
* **LLM SDK:** `openai` (抽出・DB化用軽量モデル) / `anthropic` (記事文脈制作用Claude 4.6 Sonnet)
* **Data Validation:** Pydantic
* **環境変数管理:** `python-dotenv`

## 3. データベース・スキーマ設計 (PostgreSQL + SQLAlchemy)

※ ベクトル検索を行うため、DB初期化時に `CREATE EXTENSION IF NOT EXISTS vector;` を実行します。

### 記事・トピック管理
* **`worry_seeds` (お悩みの種プールDB)**
  * `id` (UUID, PrimaryKey)
  * `original_text_hash` (String: 重複取得防止用)
  * `abstract_worry` (Text: 軽量LLMで抽象化された悩み)
  * `source_tags` (String)
  * `created_at`, `updated_at` (DateTime)
* **`blog_topics_candidates` (トピック候補DB)**
  * `id` (UUID, PrimaryKey)
  * `topic_title` (String: 提案される記事の仮タイトル・テーマ)
  * `source_data_ids` (JSONB: 発想元となった `worry_seeds.id` など)
  * `status` (String: "suggested" | "used" | "dismissed")
  * `created_at`, `updated_at` (DateTime)
* **`articles` (生成済み・投稿記事)**
  * `id` (UUID, PrimaryKey)
  * `title` (String), `content` (Text)
  * `topic_id` (UUID, ForeignKey -> `blog_topics_candidates.id`, **Nullable**: 手動テーマ入力時はNULL)
  * `status` (String: "draft" | "posted")
  * `note_url` (String, nullable)
  * `likes`, `comments` (Integer, default=0)
  * `score` (Float, 反響から算出された独自スコア)
  * `used_knowledge_ids` (JSONB: 記事生成時に使われたナレッジUUIDの配列)
  * `theme` (String: 生成時のトピックテーマ)
  * `created_at`, `updated_at` (DateTime)

### RAG・ベクトル検索用（基礎資料）
* **`knowledge_astrology` (基礎占術・運命論DB)**
  * `id` (UUID, PrimaryKey)
  * `astrology_type` (String: "西洋占星術", "タロット" など)
  * `theme_or_symbol` (String: "水星逆行", "愚者" など)
  * `fatalistic_meaning` (Text: 運命論的な解釈・悩みの肯定に使うロジック)
  * `embedding` (Vector: `pgvector`を用いた1536次元等の埋め込み)
  * `is_seasonal` (Boolean, Default False)
  * `valid_from`, `valid_until` (String, Nullable: "MM-DD" 形式で旬の期間を指定)
  * `created_at`, `updated_at` (DateTime)
* **`knowledge_mindfulness` (マインドフルネス・思想DB)**
  * `id` (UUID, PrimaryKey)
  * `philosophy_concept` (String: "マインドフルネス", "自己受容" など)
  * `encouragement_text` (Text: 温かさ、自己肯定を促すテキスト)
  * `embedding` (Vector: 類似検索用)
  * `created_at`, `updated_at` (DateTime)

### コア・ナレッジ（コピー＆心理）
* **`knowledge_persona` (占い師ペルソナDB)**
  * `id`, `trait_name` (String: "ユーモア", "お姉さん" 等)
  * `trait_description` (Text: プロンプトに埋め込む口調やスタンスの指示)
  * `is_active` (Boolean: 生成時に適用するか)
  * `priority_order` (Integer, Default 0: 小さい順にシステムプロンプトへ結合し複数特性をブレンド)
  * `created_at`, `updated_at` (DateTime)
* **`knowledge_empathy` (共感・傾聴・カウンセリング)**
  * `id`, `abstract_situation` (String: 抽象化された悩み), `empathy_approach` (Text), `warmth_level` (Integer)
  * `embedding` (Vector: 類似検索用)
  * `created_at`, `updated_at` (DateTime)
* **`knowledge_reframing` (視点の転換)**
  * `id`, `negative_state` (String), `positive_perspective` (Text)
  * `embedding` (Vector: 類似検索用)
  * `created_at`, `updated_at` (DateTime)
* **`knowledge_action` (癒やしのスモールステップ提案)**
  * `id`, `situation` (String), `small_step_proposal` (Text: 具体的な行動提案)
  * `created_at`, `updated_at` (DateTime)
* **`knowledge_copywriting` (コピーライティング)**
  * `id`, `structure_type` (String: "Introduction" | "Storytelling" | "Relief"), `suggested_text_pattern` (Text)
  * `created_at`, `updated_at` (DateTime)

## 4. APIエンドポイント実装詳細

### 1. 資料解析・RAG基盤エンジン
* **`POST /api/documents/parse`**
  * **処理:** 受け取ったPDF・Markdown・テキストをLangChain等を用いて適切な長さ（例: 500トークン単位）にチャンク分割。OpenAIの `text-embedding-3-small` 等でベクトル化し、基礎データとしてDBへインサートする。

### 2. トピック自動提案
* **`GET /api/topics/candidates`**
  * **処理:** `blog_topics_candidates` からステータスが `suggested` の一覧を返す。候補が枯渇している場合は、軽量LLM（GPT-4o-mini等）が `worry_seeds` の直近の抽象化悩みデータを参照し、新しく3〜5件のトピックタイトルを自動生成して `blog_topics_candidates` に登録する処理をキックする。

### 3. 一次資料のナレッジ抽出エンジン
* **`POST /api/extract`**
  * **処理:** ユーザーが入力した参考テキストを軽量LLM（GPT-4o-mini等）に渡し、低コストでFunction Calling (Tool Use) またはJSONモードを駆使して「構造化された悩み」「共感手法」「コピーの型」としてパースする。PII（個人情報）フィルタもここで処理する。
* **`POST /api/knowledge`**
  * **処理:** フロントエンドでユーザーがプレビュー確認・修正したJSONを受け取り、該当のナレッジDBへ永続化する。

### 4. 記事生成メインエンジン（運命論×マインドフルネス×ペルソナ）
* **`POST /api/generate`**
  * **Req:** `{ "topic_id": "...", "theme": "...", "target": "..." }`
  * **内部処理フロー:**
    1. **RAG Retriever:** `theme` と紐づく悩みのベクトル値を算出し、 `knowledge_astrology`、`knowledge_mindfulness`、`knowledge_empathy`、`knowledge_reframing` に対してpgvectorによる類似検索を実行。最適な運命論、マインドフルネス、共感、リフレーミングを取得。また `knowledge_action` に対しても状況ベースで最適な自愛スモールステップを1〜2件取得する。
    2. **Context Builder:** `knowledge_persona` の `is_active=True` の設定、`knowledge_copywriting` の型、ガードレール制約、そして**1**で引いたRAG文脈を全て1つの「System Prompt」として結合。
    3. **LLM Generation:** 高精度LLM（Claude 3.5 Sonnet等）を利用し、プロンプトエンジニアリングに基づいて高品質なマークダウン記事を長文生成。処理タイムアウトとUX低下を防ぐため、FastAPIの `StreamingResponse` (Server-Sent Events) を用いてフロントエンドへ逐次ストリーミング返却する。

### 5. 記事再生成エンジン
* **`POST /api/articles/{article_id}/regenerate`**
  * **Req:** `{ "change_persona": true/false, "custom_theme": "..." }` (省略可)
  * **処理:** 元記事の `topic_id`（または `theme`）を再利用して `POST /api/generate` と同等のRAGフローを実行。`change_persona=true` の場合は `priority_order` を1つずらして別の特性ブレンドを試みる。記事の `status` を `'draft'` に戻し新たな内容で上書き。
  * **Res:** `text/event-stream` でストリーミング返却。

### 6. フィードバックと自己進化
* **`PATCH /api/articles/{article_id}/feedback`**
  * **処理:** noteへの投稿結果（スキ、コメント数）を受け取り、スコアを更新。記事生成時に使われた `used_knowledge_ids` に対しても使用実績スコアとしての加点やランキング計算をバックグラウンドJobで実行。

### 6. SNSデータ自動収集エンジン
* **`POST /api/collect/note`**
  * **処理（全4ステップを1つの非同期バッチとして完結）:**
    1. **RSS取得・期間フィルタ:** `https://note.com/hashtag/[ハッシュタグ]/rss` から「過去1週間〜1ヶ月」の記事を取得。
    2. **プレスクリーニング:** 軽量LLMで「未解決の悩みか？（yes/no）」を分類し `no` を除外（数十トークン/件の低コスト）。
    3. **抽象化→`worry_seeds`保存:** 通過した記事をGPT-4o-miniで固有名詞排除・JSON化して `worry_seeds` へ保存（`is_used=False`）。
    4. **トピック自動生成:** 同バッチ内で `worry_seeds` → `blog_topics_candidates` 変換まで完了（`is_used=True` に更新）。`GET /api/topics/candidates` は既存データを返すだけにしレイテンシを排除する。
    * 生データは破棄する。
  * **コストガードレール:** 1回あたり最大実行件数制限、長文のトークン強制カット処理、安価モデル（GPT-4o-mini）への固定など、LLMコスト爆発防止ロジックをシステム層で担保する。

## 5. 開発フェーズ実装計画

1. **フェーズ 1: RAGとデータモデルの土台作り (Backend Core)**
   * PostgreSQLの立ち上げと `pgvector` 拡張の有効化。
   * SQLAlchemyとAlembicのエコシステム構築、各種テーブル作成。
   * OpenAI APIを用いたEmbedding（埋め込み）生成ユーティリティの実装。
2. **フェーズ 2: 占い・マインドフルネス生成エンジン (LLM Pipeline)**
   * トピックの自動生成ロジック (`/api/topics/candidates`)の実装。
   * ワンクリックお悩み自動収集バッチ (`POST /api/collect/note`) およびガードレール処理の実装。
   * RAGを組み込んだ `POST /api/generate` のスーパープロンプト（コンテキストビルダー）の構築と動作テスト（ガードレール検証含む）。
3. **フェーズ 3: フロントエンド構築 (Frontend UI)**
   * `npx create-next-app` からTailwindセットアップ。
   * APIの各種ラッパー作成と、ZustandまたはSWRを使った状態管理の実装。
   * 占いアプリらしい「温かみのある・落ち着いたUIトーン（配色の設定など）」でのページレイアウト作成。
4. **フェーズ 4: ドキュメントパイプライン＆運用フィードバック実装**
   * 管理画面から資料をアップロードしてRAG用ベクトルDBに流し込む画面実装。
   * 生成された記事の管理と、フィードバックスコア更新機能の結合テスト。
