# AI占いブログエンジン 実装ステップ

本ドキュメントはアプリケーションを非常に細かな単位に分割し、確実に動作を確認しながら進めるための実装手順書（Back-end ＆ Front-end）です。

---

## 🟢 バックエンド実装フェーズ

### Step 1: LLM基盤と共通モジュールの設定
- [x] **1.1** `app/core/config.py` - Pydanticの `BaseSettings` を用いて、`.env`（OpenRouter設定など）の厳密な型定義と読み込みを行う。
- [x] **1.2** `app/core/llm.py` - OpenRouterへの非同期クライアント (`AsyncOpenAI`) のインスタンス化と、ヘルパー関数（再試行処理、JSON出力強制）の定義。
- [x] **1.3** `app/schemas/common.py` - Pydanticを用いたAPIレスポンスの共通ベースモデルやエラーハンドリング定義を作成する。

### Step 2: 自動収集パイプラインの実装 (note RSS)
- [x] **2.1** `app/schemas/collection.py` - 収集リクエストやLLMから返ってくる一次解析結果のスキーマ定義。
- [x] **2.2** `app/services/collector.py` (RSS Parser) - `feedparser` を使い note の指定ハッシュタグRSSから指定期間分の記事を取得する処理を実装。
- [x] **2.3** `app/services/collector.py` (LLM Screening) - 取得したテキストデータを軽量LLMに渡し、「未解決の悩みか？」を判定・抽象化データをJSONで返す構造化出力を実装。
- [x] **2.4** `app/api/collect.py` - `/api/collect/note` エンドポイントを定義。取得→判定→ `worry_seeds` テーブルへの保存 までを一本の非同期フローとして実装。

### Step 3: トピック自動提案の実装
- [x] **3.1** `app/schemas/topic.py` - トピックの入出力用スキーマ。
- [x] **3.2** `app/services/topic_generator.py` - `worry_seeds` から未使用(`is_used=False`)のデータを取得するクエリ実装。
- [x] **3.3** `app/services/topic_generator.py` - 未使用データ群から、ブログテーマの仮タイトル案をLLMで自動生成し `blog_topics_candidates` に保存するロジック。
- [x] **3.4** `app/api/topics.py` - `/api/topics` の CRUD エンドポイント定義（自動生成キックと、既存一覧の取得）。

### Step 4: RAG基盤：埋め込み(Embedding)と汎用CRUD
- [x] **4.1** `app/services/embedding.py` - 任意のテキストを受け取り、`text-embedding-3-small` 相当のベクトル（OpenRouter等経由/あるいは専用モデル）を生成して返すユーティリティ。
- [x] **4.2** `app/schemas/knowledge.py` - 全7柱（占星術、マインドフルネス等）のナレッジ登録用・返却用Pydanticスキーマ。
- [x] **4.3** `app/api/knowledge.py` - テーブル数が多いので、柔軟なCRUDエンドポイント（GET, POST, PATCH, DELETE）をテーブル横断あるいは専用パスで実装。テキスト登録時に **4.1** のベクトル変換を自動で挟む機構。

### Step 5: ナレッジ一次抽出AI (Extract API)
- [x] **5.1** `app/schemas/extract.py` - 自由入力テキストからデータを抽出するリクエストスキーマ。
- [x] **5.2** `app/services/extractor.py` - 自由入力をLLMに渡し、個人情報を伏せた上で「どのナレッジに該当するか」のカテゴリ判定と JSON化 を行うロジック。
- [x] **5.3** `app/api/extract.py` - `/api/extract` エンドポイントを実装。

### Step 6: 記事生成エンジン (Generate & RAG Retriever)
- [x] **6.1** `app/services/retriever.py` - 検索クエリ（テーマ文字列）のベクトル化と、`pgvector` コサイン類似度を用いた `knowledge_astrology` (季節フラグ考慮), `knowledge_mindfulness`, `knowledge_empathy`, `knowledge_reframing` からの上位N件取得ロジック。
- [x] **6.2** `app/services/prompt_builder.py` - `knowledge_persona` のアクティブ行や `knowledge_copywriting` の構造を結合し、システムプロンプトとして文字列を構築するビルダー。
- [x] **6.3** `app/services/generator.py` - 6.1と6.2を結合し、高精度LLM（Claude 3.5 Sonnet相当）のストリーミング非同期ジェネレーター (`yield`) を実装。
- [x] **6.4** `app/api/generate.py` - `/api/generate` のエンドポイント。`StreamingResponse` を使用してフロントへSSR（Server-Sent Events）の形式で流しつつ、完了時に `articles` に結果をINSERT。

### Step 7: 再生成・フィードバック・分析ルーター
- [x] **7.1** `app/api/articles.py` (CRUD) - `articles` の一覧取得、ステータス変更。
- [x] **7.2** `app/api/articles.py` (Regenerate) - `/api/articles/{id}/regenerate` エンドポイント。元のトピックを使用し再RAG・再生成を行うストリーミング実装。
- [x] **7.3** `app/api/articles.py` (Feedback) - SNSでの反響（スキ・コメント数）を受け取り `score` を更新するPATCH通信。
- [x] **7.4** `app/api/analysis.py` - 使用されたナレッジIDの成績を集計し、`analysis_results` に反映・取得するエンドポイント。

---

## 🔵 フロントエンド実装フェーズ (Next.js)
バックエンドの各機能が完了するたびに、疎通確認を兼ねてフロントエンドを段階的に構築します。

### Step 8: フロントエンド基盤・UIコンポーネント
- [ ] **8.1** APIクライアント (`src/lib/api.ts`) の基本設定。SWRのフェッチ関数など。
- [ ] **8.2** 共通コンポーネント群（Button, Card, Modal, Input, TextArea, Badge 等）の TailwindCSS での実装。
- [ ] **8.3** レイアウト (`app/layout.tsx`, Sidebar) の作成。

### Step 9: 収集・抽出フロー UI
- [ ] **9.1** `/collect` - noteハッシュタグ収集の実行ボタンとログ（worry_seeds一覧）を表示するダッシュボード。
- [ ] **9.2** `/extract` - 手動での書籍テキスト入力・抽出結果（JSON）のプレビュー、修正用UI。

### Step 10: ナレッジ管理・生成指示 UI
- [ ] **10.1** `/knowledge` - 各ナレッジ（7柱）の一覧、追加、編集、削除（CRUD）タブ画面。
- [ ] **10.2** `/generate` - トピック（`blog_topics_candidates`）の一覧と、記事を生成するスタート画面。
- [ ] **10.3** `/articles/[id]` - ストリーミング受信によるリアルタイム文字表示（SSEコンシューマ）と、markdownレンダリング (`react-markdown`)。再生成ボタンの実装。

### Step 11: ダッシュボード・分析 UI
- [ ] **11.1** `/articles` - 生成済み記事の一覧・お気に入り管理。
- [ ] **11.2** `/` (Home) - 記事スコア（フィードバック）を通じた自己進化の現状表示と、一番貢献しているナレッジのランキング表示。
